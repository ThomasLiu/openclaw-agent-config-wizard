# 可安装 Skills（skills.sh 生态）

本 skill 向用户推荐 **已存在** 的扩展能力时，必须走检索，**禁止臆造**包名或 URL。

## 与 find-skills 的关系

在推荐「从生态安装」的 skill 前，**先按 `find-skills` skill 的流程操作**（或等价地执行其文档中的步骤）：

1. 根据用户需求提炼 **英文或关键词检索串**（可试 2～3 个同义词查询）。
2. 在可执行命令的环境里运行：`npx skills find <query>` 
 也可浏览目录：https://skills.sh/
3. **只**把检索结果里出现的 **`owner/repo@skill-name`**、`npx skills add ...`、skills.sh 链接写进方案。
4. 若 **零结果**：如实说明；可建议用户自建 `npx skills init` 或改为 **工作区自定义** `skills/<name>/SKILL.md`（并标明为自定义，不是生态包）。

## 同质化结果怎么选

当多条结果领域重叠（例如多个 React / PR review / 测试类 skill）时：

- 对照用户已说的 **技术栈**（Next vs 纯 React、Vitest vs Jest）、**语言**、**渠道**（是否偏文档/偏代码）、**风险偏好**（是否要官方维护源）。
- **主推 1 个**：用两三句说明为何最贴当前上下文。
- **备选 0～1 个**：说明与主推的差异（更重安全审计、更重脚手架等）。
- 不要为了凑数列一堆；避免让用户装功能重复的 skill。

## 与工作区自定义 skill 的边界

| 类型 | 何时用 | 方案里怎么写 |
|------|--------|----------------|
| 生态 skill | 检索命中且匹配任务 | 真实 `npx skills add owner/repo@skill` + 链接 |
| 工作区 skill | 流程/术语完全用户私有、检索无合适包 | `workspace/skills/<dir>/SKILL.md`，标注 **自定义** |

不要把「听起来合理」的生态包名写进配置草案；**无检索结果则无生态推荐**。
